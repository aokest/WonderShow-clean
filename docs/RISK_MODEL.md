# 风险模型

## 产品风险

- 误触翻页会直接打断演讲节奏
- 权限未授权会让用户误以为产品失效
- 设备连接不稳定会影响预览与录制准备
- 录制中切换源、切布局、暂停/继续等操作如果只改变 UI 状态而没有进入 manifest，会导致监视器与导出视频不一致
- 麦克风启动瞬态或停止 flush 不完整会造成开头爆音、短音轨或导出视频无声

## 技术风险

- Vision 单帧结果抖动，导致姿态分类不稳定
- 主线程负载过高，导致预览卡顿和手势延迟
- 不同摄像头视角差异大，单一阈值不通用
- `CMSampleBuffer` / `CVPixelBuffer` 等媒体帧对象跨 actor 或跨队列发送会触发 Swift 6 并发风险
- localhost 服务仍是攻击面，必须避免裸奔 HTTP、通配 CORS 和生产调试遥测
- Debug 包不等于可分发包，发布前必须确认 Release 构建、正式签名、公证和权限声明
- 录制源热键如果与系统或演示软件快捷键冲突，可能打断演讲或切错窗口
- 用户自定义 `⌘1` 到 `⌘6` 源位需要持久化但不能写入隐私敏感窗口内容

## 交互风险

- 将调试信息直接暴露给普通用户，形成认知负担
- 手势动作过多，用户难以记忆且难以形成稳定习惯
- 缺少“当前是否可触发”的可视反馈，导致盲操作
- 录制中快捷切源必须给出明确反馈，否则讲者无法确认监控台当前源是否切对

## 风险缓解

- 采用两阶段交互：先解锁，后执行
- 加入中央热区与冷却期
- 在 UI 中明确展示识别状态、可触发状态和最近动作
- 为高风险操作保留按钮兜底，不强依赖手势
- 录制主链路变更必须补测试：program 音轨时长、PiP/layout keyframe、源切换尺寸一致性、导出可播放性
- 快捷切源设计必须同时提供 UI 选择、源位编号显示和热键反馈，不只依赖键盘记忆

## 回滚策略

- Dashboard 保持单文件回滚能力
- 手势稳态控制与底层检测解耦，可单独关闭
- 识别异常时自动退回“仅预览 + 手动控制”
- 本地 HTTP 服务使用一次性 token，Release 包关闭 DEBUG-only telemetry，项目导入做大小和 schema 校验
- 摄像头帧只在 capture queue 内读取；跨 actor 只传 `Data`、`HandPoint`、manifest 值对象等 Sendable 数据
- 如果快捷切源或时间轴编辑引入不稳定，保留当前稳定的“录制源弹窗 + manifest keyframe + 手动导出”主链路作为回滚基线
