# [OPEN] gesture-regression-loop

## 症状

- 指枪翻页很不灵敏，偶发不知道怎么就翻页
- 双手八字容易误触翻页，但很难触发缩小，几乎触发不了放大
- 其他手势乱挥中偶尔翻页，无法稳定复现

## 本轮假设

- H1: 双手八字在进入离散手势识别前，没有被稳定判成缩放候选，导致落入翻页链路。
- H2: MediaPipe 对 `Victory` / `Pointing_Up` / `unknown` 的分类在真实帧中抖动，导致双手缩放姿态覆盖率不足。
- H3: `StreamingGestureRecognizer` 在双手场景下仍可能从某一只手提取到水平位移，从而误发翻页。
- H4: 当前平滑/样本窗口参数让指枪有效位移被削弱，导致翻页需要更长时间才触发，看起来“不灵敏”。
- H5: 缩放命令其实已经发出，但 HTML 测试页没有连续收到或正确应用。

## 计划

1. 只添加调试插桩，不改业务逻辑
2. 让你按固定步骤复现一次
3. 对照日志判断是“识别前半段问题”还是“命令投递问题”
4. 只做最小修复

## 证据

- 日志中几乎全部为 `CameraPreviewService.handleNoHandsDetected` 且 `engine = Vision 增强版`
- 本次复测日志里没有出现 `engine = MediaPipe Sidecar`
- 本次复测日志里没有出现任何 `pointsCount = 2` 的 `frame processed`
- HTML 页面只收到翻页命令，未收到任何缩放命令

## 结论

- H1: 成立。双手八字没有稳定进入缩放候选，根因是上游没有稳定提供双手点位。
- H2: 部分成立，但更上游的问题更明显：MediaPipe 在本轮复测里没有稳定接管，日志几乎完全落在 Vision 增强版。
- H3: 成立。因为输入长期退化成单手，离散识别只能持续输出 `swipeLeft` / `swipeRight`。
- H4: 不成立。本轮主要不是“指枪位移被削弱”，而是识别后端退化导致输入条件不对。
- H5: 不成立。翻页命令从控制器到 HTML 页面应用链路完整可见。

## post-fix 追加证据

- 修复“空帧不再整体切回 Vision”后，日志里重新出现了很多 `pointsCount = 2`
- 但双手八字窗口里仍频繁出现 `zoom candidate evaluated` 且 `prioritized = false`
- 甚至在 `startShapes = lShape,lShape`、`endShapes = lShape,lShape`、`lShapeCoverage = 1` 时，仍被判成 `false`

## post-fix 追加结论

- 双手缩放当前卡在“候选判定过严”，不是卡在页面应用层
- 最小修复应为：当最近窗口已稳定出现双手八字时，直接优先进入缩放链路，停止依赖过严的运动阈值

## post-fix 进一步证据

- 双手缩放链路已经开始稳定发送 `setZoom`，说明桥接和连续缩放主链路已经打通
- 但缩放目标值存在 `107% -> 112% -> 124% -> 135% -> 143%` 这类短时间大跳变，导致页面看起来浮夸和晃动
- 双手八字窗口中仍出现翻页命令，说明离散翻页识别在缩放姿态期间没有被压住
- `swipeLeft` 比 `swipeRight` 更常在更长样本窗口里才被识别，导致“上一页”主观上略不稳定

## 本轮修复

- 为连续缩放增加“单次最大缩放步长”限制，避免短时间过大跳变
- 在双手缩放姿态稳定出现时，直接抑制离散翻页识别
- 将单手步进缩放从 15% 下调到 8%，同步降低 HTML 测试页的离散缩放步长

## 继续追加证据

- 当前“抑制离散翻页”的实现只看 `gestureSamples` 中已有窗口，没有把正在判断的“当前帧”一起纳入
- 这会导致双手八字刚稳定形成时，仍有 1 帧左右的机会先落入离散翻页链路
- 用户主观上看到的“缩放开始有了，但还是偶尔误翻页”，与这类延后一拍的门控缺口一致
- HTML 测试页当前使用速度型弹簧动画，即使 Swift 侧目标缩放已经变小，页面视觉上仍会显得晃和浮夸

## 继续最小修复

- 抽出纯策略 `DiscreteGestureSuppressionEvaluator`，把“当前帧 + 最近窗口 + zoomPoseStreak` 一起纳入双手缩放抑制判断
- 将连续缩放从固定单次步长上限改为“细动作小步长、快动作大步长、整体仍受最大步长约束”的动态步长
- 将 HTML 测试页的缩放/平移动画从弹簧速度模型改为分段缓动，减少过冲和回摆
- 小幅提前 `StreamingGestureRecognizer` 的离散判定窗口，尝试缓解“上一页比下一页更慢半拍”的体验

## 新一轮用户反馈

- 下一页已经稳定
- 上一页仍不如下一页稳定
- 双手八字误翻页已接近消失
- 连续缩放仍偏慢，离“像 iPad 触摸屏一样顺滑”还有差距
- 用户明确质疑当前是否“只用了 1 个结构点”，这与现有 Swift 侧轻量锚点模型一致

## 本轮继续修复

- 将 MediaPipe 采样间隔从 `90ms` 下调到 `60ms`，提高连续缩放刷新频率
- 继续前移离散翻页判定参数，缓解“上一页总慢半拍”
- 将 `Pointing_Up` / `Victory` 从单点锚点改为多 landmarks 加权锚点
- 继续提高 `ContinuousZoomTracker` 的细动作响应与大动作跟手速度

## 最小修复

- 不再因为 MediaPipe 连续空帧就整体切回 Vision
- MediaPipe 空帧只记为一次未识别，下一帧继续尝试 MediaPipe
- 保留 `frame == nil` 时的健康检查回退逻辑
