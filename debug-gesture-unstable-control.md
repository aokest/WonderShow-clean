# Debug Session: gesture-unstable-control [OPEN]

## 症状

- 用户反馈：`已经是0.5，新ui。手势还不稳，但是可以操控。`
- 当前状态：应用已能启动并进入新 UI，手势链路可工作，但稳定性与缩放精度不足。

## 预期

- 左右翻页应在自然手势下稳定触发，不应明显漏触或误触。
- 双手缩放应连续、平滑、可复现，不应忽大忽小或偶发一次后难以复现。

## 可证伪假设

1. MediaPipe 仍然存在较多空帧或双手帧不连续，导致状态机输入抖动。
2. Vision fallback 与 MediaPipe 在切换时造成手势源不一致，导致控制不稳。
3. 缩放跟踪器对双手距离变化过于敏感或重置过快，导致“有反应但不精确”。
4. 翻页和缩放共用同一采样窗口，双手乱动时会把缩放误判成翻页或互相干扰。
5. 个性化校准残留、解锁窗口或冷却窗口参数仍不适合当前摄像头视角，导致漏触发。

## 计划

1. 读取现有调试日志，判断是“检手不稳”还是“状态机参数不稳”。
2. 检查当前手势链路代码中与双手缩放、fallback、窗口参数相关的实现。
3. 如证据不足，只增加最小插桩。
4. 基于证据实施最小修复，并给出可复现验证步骤。

## 证据记录

- E1: 旧调试日志末尾仍出现大量 `no hands detected`，并且在 `MediaPipe Sidecar` 与 `Vision 增强版` 间切换，说明输入帧稳定性仍然不足。
- E2: 新一轮稳定性日志文件未生成，说明用户复测时很可能仍运行旧进程，因此本轮新增插桩没有真正写入。
- E3: 代码证据显示 `CameraPreviewService.processDetectedPoints()` 中，只要 `isLikelyTwoHandZoomCandidate()` 为真就会提前 `return`；但 `handleContinuousZoom()` 在当前手型不支持缩放时会返回 `false`。
- E4: 旧实现的 `isLikelyTwoHandZoomCandidate()` 只要求双手“不是 fist”且方向相反，不要求两只手都满足 `allowsZoom`，因此普通双手摆动也可能误入缩放候选分支。

## 结论

- H1 成立：输入帧仍有空帧，尤其双手场景下不稳定。
- H3/H4 成立：连续缩放候选分支过宽，并且失败后仍提前返回，会吞掉后续的普通翻页识别。
- 最小修复方案：
  1. 将“连续缩放候选”提炼为纯策略 `ContinuousZoomCandidateEvaluator`。
  2. 候选判定要求起止两帧的双手都满足缩放手型。
  3. 只有 `handleContinuousZoom()` 真正接管时才提前返回，否则继续走后续离散手势识别。
  4. 画面中出现多只手时，只选择热区内更靠近画面中心的最多两只手作为输入。
  5. 缩放过程中在每次输出后重置基线，并在手部距离变化很小的连续帧后自动重置基线，减少“停手后还在继续缩放”的体感。
